title: "Service Credentials for Unattended Automations and Scripts"
sidebarTitle: "Service Credentials"
description: "Create a static bearer-token key for scripts, CI jobs, and server integrations that need to reach a Cortex brain without a human approving in a browser."
---

Most connections sign in through your browser: you approve, and the agent receives a key scoped to one brain. But some things have no human available to click approve: a scheduled script, a server-to-server integration, a CI job. For those cases, Cortex issues a **service credential** -- a key you create yourself and paste directly into the automation.

A service credential is still bound by exactly the same rules as any other connection. It belongs to one brain, carries one access level, and opens nothing else. It simply skips the browser approval step.

## Creating one

<Steps>
  <Step title="Open the brain's credentials page">
    From the brain's **Connect** page, follow the link labelled "Automation without a human?" to reach the service credentials section. Alternatively, go to the brain's settings and find **Service credentials**.
  </Step>

  <Step title="Issue a credential">
    Give it a label you will recognise later -- for example "nightly import" -- then choose its access level (read, or read and write) and create it. Cortex shows you the key, which starts with `ck_`, exactly once. Copy it now; you will not see it again.
  </Step>

  <Step title="Use it">
    Send the key as a bearer token in the `Authorization` header of your requests to the Cortex API. For example, pushing a note into the brain's inbox:

    ```bash
    curl https://api.oncortex.ai/api/v1/inbox \
      -H "Authorization: Bearer ck_your_key_here" \
      -H "Content-Type: application/json" \
      -d '{"filename": "nightly.md", "content": "...", "externalId": "run-2026-07-08"}'
    ```
  </Step>
</Steps>

## Keeping it safe

<Warning>
A service credential is a password. Store it in your automation's secret store and never commit it to code or share it in plain text.
</Warning>

- The credential only reaches its one brain, at its one access level. A read-only credential cannot write, and no credential can reach a different brain.
- Revoke it at any time from the same credentials page. Revocation takes effect immediately. You can then issue a fresh credential if you need to rotate the key.

## When to use the normal connection flow instead

If a human is present and the tool has a browser, use the standard [connection flow](/connect/how-connections-work) rather than a service credential. It is simpler to set up and straightforward to revoke per agent. Service credentials are specifically for the unattended case where no browser approval is possible.

For the full list of things you can do via the API, see the [API reference](/reference/api).
